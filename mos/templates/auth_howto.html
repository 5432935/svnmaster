SVN 权限指引

概念

认证(Authentication）和授权(Authorization）

这两个术语经常一起出现。其中认证的意思就是鉴别用户的身份，最常见的 方式就是使用用户名和密码，授权就是判断用户是否具备某种操作的权限，在Subversion 里提供了passwd文件作为认证文件，authz-db 文件作为授权文件。

svnserve 下的配置文件

假设使用 svnserve 作为SVN服务，一个版本库的物理结构为：

D:\SVNROOT\PROJECT1
├─conf
├─dav
├─db
│ ├─revprops
│ ├─revs
│ └─transactions
├─hooks
└─locks

其中 conf 下面有三个文件：

authz
passwd
svnserve.conf

svnserve.conf 是这个版本库的配置文件，当使用 svnserve 时，这个配置文件决定了使用什么认证和授权文件：

[general]
anon-access = none
auth-access = write
password-db = passwd
authz-db = authz
realm = My First Repository

上面的配置说明：

  1. 匿名用户（anon-access）的权限为 none(可以为 none、read、write）
  2. 认证用户（auth-access）的权限为 write(可以为 none、read、write。
  3. SVN 的权限认证方式是基于此处的设置和 anthz 授权文件中的设置，两者都满足(& 方 式 ) 才 成 立 ， 即read&read=read,read&write=read,write&write=write, 所 以 此处应把认证用户的权限设置为 write，否则 authz 文件中关于写权 限的设置都会无效的
  4. 使用了 “svnserve.conf”同目录的 passwd 和 authz，其中的 password-db 指定了用户密码文件，authz-db 是授权文件。
  5. realm 为“My First Repository” ，这个值作为客户端保存服务器端信息(如用户和密码)的唯一识别符

注意：使用 Apache 作为服务器时，根本就不会使用“svnserve.conf”文件的内容，而是会使用 Apache 自己的配置。

基于 svnserve 的多版本库权限布局

使用 svnserve 时，为了管理的方便，应该使用相同的认证和授权文件，所以应该让所有版本库的配置文件 svnserve.conf 指向同一个 password-db 和authz-db 文件。下面是一个多版本库的目录：

D:\SVNROOT
├─project1
│ ├─conf
│ ├─dav
│ ├─db
│ │ ├─revprops
│ │ ├─revs
│ │ └─transactions
│ ├─hooks
│ └─locks
└─project2
├─conf
├─dav
├─db
│ ├─revprops
│ ├─revs
│ └─transactions
├─hooks
└─locks

D:\SVNROOT 下有两个目录 project1 和 project2， 都已经创建了版本库，所 以 我 们 修 改 每 个 conf 目 录 下 的 svnserve.conf ， 使 之 指 向 同 一 个password-db 和 authz-db 文件。

password-db = ..\..\passwd
authz-db = ..\..\authz

这样，D:\SVNROOT\passwd 和 D:\SVNROOT\authz 就控制了所有版 本库的 svnserve 访问。

另外在后面的操作中要关闭匿名访问，应该去掉 “anon-access = none”前的“#”号，保证只有认证用户可以访问。

注意：还有一点需要注意，那就是 svnserve 的“realm”的值，在上面的设置下，应该保证所有的版本库使用相同的 realm 值，这样，对版本库的密码缓存可以在多个版本库之间共享。

基于 Apache 的多版本库权限布局

使用Apache作为SVN服务时，配置文件内容如下：

<Location "/svn/">  
DAV svn  
SVNParentPath "D:\RDMSP\SVN_Repos"
SVNListParentPath on
SVNIndexXSLT "/svnstyle/svnindex.xsl"
AuthType Basic
AuthName RDMSP
AuthUserFile "D:\RDMSP\SVN_Repos\RDMSP_svn_passwd"
Require valid-user  
AuthzSVNAccessFile "D:\RDMSP\SVN_Repos\RDMSP_svn_authz"
</Location>

其中，AuthUserFile 和 AuthzSVNAccessFile 分别指出了 认证文件 和 授权文件的位置。<Location "/svn/"> 目录下所有配置库的权限配置均在这两个文件中配置。

权限继承与冲突

一般而言，下层目录的权限会继承上层目录的权限，如用户dev有/codes的权限，则自动会有 /codes/trunk的权限，但同时，SVN权限遵循 特殊优先于一般 准则，即特别配置的权限优先于自动继承的权限，如：

[/]
dev = rw
test = 

[/codes/trunk/1.0.0]
dev=r
test = rw

则用户 dev 只有 /codes/trunk/1.0.0 读权限而没有写权限;

用户 test 虽然无法看到 / 的内容，但可以对 /codes/trunk/1.0.0 进行读写

还有另外一种情况：

一个用户 通过 不同的组 获得了一个路径下不同权限时，采用的优先级是禁止 > 只读 > 读写，如:

group_dev = dev,test
group_test = test

[/codes/trunk/1.0.0]
@group_dev = rw
@group_test = r

则 用户 test 只有 /codes/trunk/1.0.0 的读权限而没有写权限。

案例

我们以一个实际的案例来说明权限配置

  1. 前提：版本库禁止任何匿名用户的访问，只对认证用户有效。

  2. 设置以下用户和组：

    • admin：配置管理管理员，对版本库有完全的管理权限。
    • p1_admin：project1 的管理员，对 project1 有完全权限。
    • p1_dev：project1 的开发者，对 project1 的 trunk 有完全的权限，但是对其中的/trunk/admin 目录没有任何权限。
    • p1_test：project1 的测试者，对 project1 的 trunk 有完全的读权限，但是对其中的/trunk/admin 目录没有任何权限。
    • p2_admin：project2 的管理员，对 project2 有完全权限。
    • p2_dev：project2 的开发者，对 project2 的 trunk 有完全的权限，但是对其中的/trunk/admin 目录没有任何权限。
    • p2_test：project2 的测试者，对 project2 的 trunk 有完全的读权限，但是对其中的/trunk/admin 目录没有任何权限。

  3. 对应的组及组的用户：

    p1_group_admin:p1_admin
p1_group_dev:p1_dev
p1_group_test:p1_test
p2_group_admin:p2_admin
p2_group_dev:p2_dev
p2_group_test:p2_test

4、修改 D:\SVNROOT\passwd 文件

用户和密码在 D:\SVNROOT\passwd，我们为每一位用户设置权限，文件内容如下：

[users]
# 标示用户信息开始
p1_admin = p1_admin
p1_dev = p1_dev
p1_test = p1_test
p2_admin = p2_admin
p2_dev = p2_dev
p2_test = p2_test

为了便于验证，所有密码和用户名一致，如果你使用的是其他认证方式，这 一步可能不同，但是用户名应该都是一样的。

注意：SVNserver 使用的是未加密的密码存储方式，如果使用 Apache 作为 SVN 服务器的话，需要使用 apache 自带的 htpasswd 或 htdigest 程序生成密码文件(密码加密)。

5、配置授权，修改 D:\SVNROOT\authz

[groups]
# 标示组信息开始
p1_group_admin = p1_admin
p1_group_dev = p1_dev
p1_group_test = p1_test
p2_group_admin = p2_admin
p2_group_dev = p2_dev
p2_group_test = p2_test

[/]
# 指定所有的版本库默认只读，root 可读写
# 如果只有一个版本库使用此授权文件，[/]代表此版本库的根目录
# 在使用子目录时不用再路径前面加版本库名称
# 如：[projectname:/trunk]可以直接写成[/trunk]
* = r
admin = rw

[project1:/]
# 指定对版本库 project1 根目录的权限
@p1_group_admin = rw
@p1_group_dev = rw
@p1_group_test = r

[project1:/trunk/admin]
# 指定对版本库 project1 的/trunk/admin 根目录的权限，
# p1_group_a 读写，p1_group_d 和 p1_group_t 没有任何权限。
@p1_group_admin = rw
@p1_group_dev =
@p1_group_test =

[project2:/]
# 指定对版本库 project2 根目录的权限
@p2_group_admin = rw
@p2_group_dev = rw
@p2_group_test = r

[project2:/trunk/admin]
# 指定对版本库 project1 的/trunk/admin 根目录的权限
@p2_group_admin = rw
@p2_group_dev =
@p2_group_test =

经过以上设置以后，你会发现一些有趣的事情。 当使用用户“p1dev”，检出project1 的 trunk 时，目录是空的，好像 admin 目录根本不存在一样，当使用 p1dev 用户浏览版本库时，能够看到 admin 目录，但是其中的内容却无法看到。
